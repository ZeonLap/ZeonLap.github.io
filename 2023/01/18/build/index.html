节点的数据结构

多叉树的表示方式有许多种，这里以父亲-孩子-兄弟表示法记录信息。以结构体struct Node记录每个节点的信息。具体说来，包括了以下信息：

这里，以整数记录每个节点的亲缘信息。即每个节点的父亲、兄弟等信息都以整数而不是指针的方式记录。这是因为题目所示多叉树的节点有其唯一的编号，即只需以Node nodes[N]储存所有节点的信息即可。故若已知某个节点的某位亲戚编号为i，只需用nodes[i]便可找到该亲戚，这大大减少了储存空间，同时也带来了一些便利（遍历时若该节点为0则知不存在）。

Task 1：建树

以邻接表的方式读入整颗多叉树的数据后，如何初步建立每个节点的数据是首先要考虑的问题。由于树高，规模等信息不便于在读入数据时直接确定，这里分两步建立每个节点的数据：

一、在读入时建立节点间的亲缘关系，即整棵树的拓扑结构。以邻接表读入数据时，可以初步建立兄弟、父子间的关系。

二、整棵树的信息读取完毕后，可以从树根（即nodes[1]）开始遍历整棵树，建立关于size，height，suffixHeight与suffixSum的信息。遍历的过程有点类似于二叉树的后序遍历，不过这里的顺序是从lastChild（幼子）开始，逐步往其兄长方向遍历，直到走到其firstChild（长子），再遍历自己本身。二叉树的后序遍历是lc->rc->x，而这里遍历多叉树的顺序是lastChild->...->firstChild->x。本题偷懒地采用递归的方式遍历整棵树，大致思路是：

\(traverse(int\ x)\ \{\)

​ \(currentNode = x.lastChild\)

​ \(while(currentNode)\)

​ \(traverse(currentNode)\)

​ \(currentNode=currentNode.prevSibling\)

​ \(set(x)\)

\(\}\)

可以看出，递归基是叶子节点，因其没有孩子。

三、可以看出，遍历时利用了set()帮忙设定节点信息。各信息的维护细节如下：

  • size：若x是叶子节点，则直接设为1。否则设为x长子的suffixSum + 1。

  • Height：若x是叶子节点，则直接设为0。否则设为x长子的suffixHeight + 1。

  • suffixHeight：设为max(x.height, x.nextSibling.suffixHeight)。

  • suffixSum：设为x.nextSibling.suffixSum + x.size。

其中suffixSum的存在并非必须，后续也不必维护，但suffixHeight的存在能将复杂度控制在\(O(cost)\)内。

Task 2：删除子树

在程序中，删除以x作为节点的子树的接口为detach(x)

删除子树的工作也可以拆分成以下几步进行：

一、改变节点x附近的兄弟姊妹关系。有几个特殊情况要考虑：x是其父的长子或者幼子。这时除了改变兄弟姊妹的信息，还需重新设置父亲的长子/幼子。（除此以外，若不考虑特殊情况，可能会改变nodes[0]的信息，这是我们不希望的）

二、自x沿直系祖先渠道改变节点的size，直至根节点（nodes[1]）。每个节点进行size -= x.size的操作

三、逐步更新节点的height与suffixHeight信息。若一个节点是作为某节点的父节点访问的，则他的height应更改为curNode.firstChild.suffixHeight + 1，suffixHeight应改为max(curNode.height, curNode.nextSibling.suffixHeight)。若一个节点是作为某节点的兄长访问的，则其height不需改变，suffixHeight如上一致的改变即可。

这里存在一个思考：将suffixHeight存储下来的目的在何处？倘若不记录suffixHeight，那么更新节点信息时，可能会带来\(O(n)\)的开销。举个例子，现删除以x为节点的子树，那么在更新x直系祖先（父，祖父，...）的高度时，需要遍历它的每一个孩子，查询他孩子中高度的最大值，并以此作为修改依据。反之，如果存储了suffixHeight，在修改高度时，该祖先不需要搜索节点x的nextSibling结点信息，也不需要搜索结点x父亲的nextSibling结点信息，也不需要搜索结点x父亲的父亲的nextSibling信息...这便是cost与复杂度存在关联的地方。维护了suffixHeight使得更改高度时不需遍历所有的孩子，遍历的“深度”由cost所限制！

四、清空x自身的亲缘关系，这是容易的。

Task 3：插入子树

将子树x作为结点d的第rank个孩子插入，接口为attach(int x, int d, int rank)。

插入子树与删除子树是类似的。唯一不同的是需要多考虑一个因素——新插入结点是目的节点的第rank各孩子。这里先把这个位置的下一个地点求出来，也有几种特殊情况：

  • x作为d唯一的孩子接入，这时x既是长子也是幼子。
  • x作为d的长子接入。
  • x作为d的幼子接入。
  • 一般的情况。

分别考虑他们带来的兄弟姊妹间的新关系，以及父子间的关系即可。更改size，height的过程与detach完全一致，耗时自然也是\(O(cost)\)。

Subtask：读入路径

题目给出的目标节点并不是直接以其标号int给出，而是给出一条路径。程序中以int getNode(int lenth, int* a)作为接口查询该路径。

当然了，比较坑的一个点在于路径可能从某个位置开始非法。我们需要记下最后一个合法的点，总体来说这里的实现是简单的。

Subtask：查询

直接输出目标节点的size或者height即可。

复杂度分析

时间复杂度

建树过程需要遍历每一个结点，复杂度为\(O(n)\)。

以下简单证明移动子树移动的时间复杂度为\(O(cost)\)：

假设每次操作的操作花费为\(COST\)，由题目有\(cost=\sum COST\)。

由前文所述，单次删除/插入子树的操作都不会涉及到子树x，或目的节点d，的弟弟结点，也不会涉及到他们祖先的弟弟结点；

故整个更改路线的长度不会超过\(COST\)。也就是说，单次操作的时间复杂度为\(O(COST)\)。两边求和，由所有操作的时间复杂度为\(O(cost)\)。

由于操作次数\(m << n \approx cost\)，得总的时间复杂度为\(O(n+cost)\)。

空间复杂度

只需储存\(n\)个结点的信息，空间复杂度为\(O(n)\)。

其他问题
