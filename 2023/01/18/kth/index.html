算法分析

采用堆（Heap）的数据结构解决这道题。

为了简明起见，先分析这道题的二维情况。假如给定了两个数组a[]，b[]，我们需要求a[x] + b[y]中第k小的那组x, y。我们可以先把数组a，b都进行排序。那么a[x] + b[y]存在一个偏序关系：可以想象一个二维n * n的网格，左上角为（1，1）。对于网格中的每个节点，其右边与下面的结点都大于他，而其左边与上面的结点都小于他。也就是说，首先我们知道（1，1）是所有节点中最小的那个；其次，与（i，j）存在偏序关系的结点且恰好只比（i，j）小的结点只有（i + 1，j）与（i，j + 1）两个。

于是，我们可以想象这么一个堆——初始只有一个元素（也是堆顶）（1，1），他是a[x] + b[y]中第1小的元素。在求第2小的元素时，我们需要先把（1，1）弹出来，把（1，2）与（2，1）加入到堆中。这时，新的堆顶是第2小的元素。以此类推，我们接下来再将刚才的堆顶（假如说是（1，2））弹出来，把（2，2）和（1，3）加入到堆中。这时这三个元素组成的堆的堆顶就是第3小的元素。重复以上的步骤k次，我们就可以动态地维护这个堆，得到第k小的那组（x，y）。

实际上，算法的正确性可以如此来理解：第k步堆中的元素都可以看作是第k小答案的候选者（candidate）。因为经历了前面的k - 1次操作，弹出的元素一定比已经存在堆中的这些元素要小。所以他们不可能比第k小还小。而且，还未加入堆中的那些元素与他们存在偏序关系，也就是说还未加入堆中的元素至少是k + 1小。而现有堆中的这些元素之间不完全存在偏序关系（有些元素之间可能不存在偏序关系），所以说他们有可能成为第k小的答案。如此即是做到了不重复，不遗漏。

堆的实现

主要仿照了讲义上的实现。借用了percolateUp与percolateDown完成堆的各种操作。

堆中元素的key是一个个三元组，代表了其对应的排好序的a[]，b[]，c[]下标。也即是上文所说的（x，y，z）。

排序

由于我们无法直接获取a, b, c三个数组的元素，而是需要借用compare函数，故我们在排序时需要固定另外两个数组的元素，对其中一个数组的元素进行排序（实际上是对下标排序）。这里使用了归并排序（MergeSort），且在参数列表中增加了一个参数表示固定的是哪两个数组的元素。

需要注意，我们排序的是下标，三元组表示的也是下标。

复杂度分析

时间复杂度

归并排序的复杂度为\(O(3nlogn)\)。查询第k小的元素需要经历k次弹出与至多3k次插入。故这一步的时间复杂度为\(\sum logk = O(klogk)\)。总的时间复杂度是\(O(nlogn + klogk)\)。

空间复杂度

数组的复杂度为\(O(n)\)，堆的复杂度为\(O(3k) = O(k)\)，总的复杂度为\(O(n + k)\)。
