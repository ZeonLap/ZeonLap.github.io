01.cpp——05.cpp

06.cpp——10.cpp

06.cpp 至 10.cpp采用分块数组的方式解决Zuma。分块数组每次查询与插入的时间复杂度都为\(O(\sqrt n)\)，总复杂度为\(O(m\sqrt n)\)。对于\(n \le 500000, m\le500000\)的数据而言，时间复杂度是合格的。

数据结构简要分析

  • char a[1 << 22]储存未分块的原始珠子序列，1 << 22 = 2^22 = 4194304 > 500000，int alen储存珠子总长度。

  • const int plen_target = 1 << 11记录分块数组每一块（每一行）的目标长度（初始最大长度），const int plen_bound = 1 << 12记录每一块的最大长度。

  • char p[1 << 12][plen_bound]记录分块后的珠子序列，size_t plen[1 << 12]记录每一块的实际长度，size_t pn记录块数，struct Rank记录某颗珠子所在的行与列数。

  • 期望在于：在插入/消除过程中，维护一个2048 * 2048的分块数组。

    • 插入某个珠子：通过find函数，在\(O(\sqrt n)\)的时间内将一维顺序与二维的行与列对应。插入操作可以看作只在该块数组上进行，时间复杂度也为\(O(\sqrt n)\)。
    • 重构分块数组：分块数组的某块可能过长，需要重构。
    • 检查最长消除序列：维护两个Rank类对象l与r，分别记录消除的起始点与（超）终止点的位置。由于稳定的序列不存在三个相同颜色的珠子相连，故每次查询最长消除序列的均摊时间为\(O(1)\)。
    • 执行消除：将l与r之间的所有珠子消除，类似于向量的remove操作。由于每块数组的长度不大于\(2\sqrt n\)，故这步操作的时间复杂度为\(O(\sqrt n)\)。

  • play中的基本操作只包括以上三种。以下具体讨论每个样例的错误时，涉及到的错误也都来自于play中这些操作的错误。

06.cpp：Wrong Answer

  • 错误原因：插入珠子时产生错误。如前所述，分块数组p的最大块容量为1 << 22 = 4096。每执行一次play操作（插入与消除），rank所在行的块长度就要加一，rank位置后本块的珠子需要向后移一个顺位。

容易看出，由于每一块的最大长度为4096，远小于操作数500000。极端情况下，倘若m次插入全集中在某一行（且没有发生消去），该块的长度必然比4096要大。然而06.cpp没有重构分块数组的操作，这使得plen[pos.first]可能大于4096。在执行memmove(cur + 1, cur, succ_len)时，由于数组p空间的连续性，cur后succ_len个位置可能已经进入下一块。而我们在进行某一块的操作时，并不想对其他块的数据进行改动，这便导致了Wrong Answer（因为在执行memmove时，我们总假定该块后面还有足够的空白空间供我们操作）。

  • 测例构造：珠子初始长度足够长，满足极端情况——在某一行插入大于5000(> 2048)个元素且没有消除。利用python脚本可以简单地构造：

  • 错误原因：检查最长消除序列时出错。检查能消除的珠子时，可能有连着的珠子分别在上下两行。故在检查时，若r.second < 0，说明检查已经退入上一行，需要令l.first--; l.second += plen[l.first];

    然而，若碰到在先前的过程中，存在某一整行被消除的情况，07.cpp便有可能出错。

    若某一整块数组被消除，该块的plen更新为0，实际上可以理解为该块已经不存在了。当该块的下一行向前寻找可消除对象时，若l.second跌入-1，则其会访问p[l.first - 1][-1]。这相当于访问了p[l.first - 2][4095]！我们想通过p[l.first - 1][-1]访问p[l.first - 2][plen[l.first] - 1]，却访问到了p[l.first - 2][4095]，这当然是错误的。
