关于SSH的使用

1. 如何使用ssh登陆服务器？

ssh username@ip -p port

2. 配置ssh免密登陆

Step 1: 生成SSH密钥对

首先，在本地计算机上生成SSH密钥对。这包括一个私钥（private key）和一个公钥（public key）。

ssh-keygen -t rsa

如果是生成github/gitlab的ssh密钥对，则要按照以下的格式：

ssh-keygen -t rsa -C "example@example.com"

系统将会提示你指定保存密钥文件的路径和文件名。默认情况下，密钥对将保存在用户主目录下的.ssh文件夹中。可以直接按回车键接受默认选项。

Step 2: 将公钥复制到目标服务器

生成公钥后，需要将公钥复制到目标服务器上。

ssh-copy-id username@remote_host
或者
ssh-copy-id myserver

其中，username是你登陆远程服务器时使用的用户名，remote_host是远程服务器的地址或域名。执行该命令后，系统会要求输入你的密码。

组合优化问题：退火算法

退火过程

  • 升温：随着温度的不断上升，粒子逐渐脱离其平衡位置，变得越来越自由，直到达到固体的溶解温度，粒子排列从原来的有序状态变为完全的无序状态。

  • 退火：随着温度的下降，粒子的热运动逐渐减弱，粒子逐渐停留在不同的状态，其排列也从无序向有序方向发展，直至到温度很低时，重新以一定的结构排列。

  • 粒子不同的排列结构，对应着不同的内能水平。如果退火过程是缓慢进行的，也就是说，温度的下降如果非常缓慢的话，使得在每个温度下，粒子的排列都达到一种平衡态，则当温度趋于绝对零度时，系统的内能将趋于最小值。

Metropolis准则

从状态i转换为状态j的转换准则：

如果$E(j)\leq E(i)$，则状态转换被接受；

如果$E(j)>E(i)$，则状态转移被接受的概率为：$\exp(\frac{E(i)-E(j)}{kT})$

Boltzmann分布律：$P_i(T)=\frac{e^{-\frac{E(i)}{kT}}}{Z_T}$

退火过程分析

1. 同一温度下两个内能不同的状态

假设$E(i)<E(j)$，那么$P_i(T)-P_j(T)>0$。也即在任何温度下，系统处于低内能状态的概率大于处于高内能状态的概率。

2.高温下的情况

$$\lim_{T\rightarrow\infin}P_i(T)=\lim_{T\rightarrow\infin}\frac{\exp(-\frac{E(i)}{kT})}{\sum_{j\in S}\exp(-\frac{E(j)}{kT})} = \frac{1}{\left| S\right|} $$

（每一项都是$1$）

也就是说，高温情况下各个状态的概率是相等的。

3. 低温下的情况

$$\lim_{T\to 0}P_i(T) = \lim_{T\to 0}\left[\frac{e^{-\frac{E(i)}{kT}}}{\sum_{j\in S}e^{-\frac{E(j)}{kT}}}\right] = \lim_{T\to 0}\left[\frac{e^{-\frac{E(i)-E_m}{kT}}}{\sum_{j\in S}e^{-\frac{E(j)-E_m}{kT}}}\right] \\ = \lim_{T\to 0}\left[\frac{e^{-\frac{E(i)-E_m}{kT}}}{\sum_{j\in S_m}e^{-\frac{E(j)-E_m}{kT}} + \sum_{j\notin S_m}e^{-\frac{E(j)-E_m}{kT}}}\right] \\ = \begin{cases} \frac{1}{\left|S_m\right|}, & i \in S_m \\ 0, & i \notin S_m \end{cases} $$

$S_m$表示系统最小内能状态的集合，$E_m$表示系统的最小内能

即系统等概率趋近于几个内能最小的状态之一（系统达到内能最小状态的概率为1），而系统处于其他状态的概率为0。

4. 温度缓慢下降的情况

$$\frac{\partial P_i(T)}{\partial T}=\frac{\partial}{\partial T}\left[\frac{e^{-\frac{E(i)}{kT}}}{Z_T}\right] \\ =\frac{P_i(T)}{kT^2}[E(i)-\overline{E_T}] \\ \begin{cases} > 0, & E(i) > \overline{E_T}, 高能状态 \\ < 0, & E(i) < \overline{E_T}, 低能状态 \end{cases} \\ 其中\overline{E_T}=\sum_{j\in S}E(j)P_j(T)为状态内能的平均值 $$

也就是说：系统处于低能状态的概率随着温度的下降单调上升，而系统处于高能状态的概率随着温度的下降单调下降。

从另一个角度说明了，当温度为绝对零度时，系统只能为内能最小的状态。

5. 退火过程的三个条件

初始温度必须足够高（各个状态都等概率）；

在每个温度下状态的交换必须足够充分；

温度$T$的下降必须足够缓慢。

模拟退火算法

  • 基本思想：在局部搜索算法中，从领域中随机选择一个解$j$，按照Metropolis准则接受该解。

算法需要解决的几个问题：

(1) 初始温度$t_0$

(2) 温度$t$的衰减函数

(3) 算法的终止准则

(4) 每个温度$t$下的马尔可夫链长度（什么叫做达到平衡）

初始温度

  • 基本原则：足够高的初始温度，以等概率处于任何一个状态
$$\exp(-\frac{\Delta f(i,j)}{t_0})=P_0 \approx 1 $$

得：

$$t_0=\frac{\Delta f(i,j)}{\ln(P_0^{-1})} $$

其中$\Delta f(i,j)$为状态$j$和状态$i$的指标函数差，可由随机产生的序列$S$计算：

$$\Delta f(i,j)=\max_{i\in S}f(i) - \min_{i\in S}f(i) $$
$$\Delta f(i,j)=\frac{\sum_{i,j\in S}|f(i)-f(j)|}{|S|^2} $$
$$\Delta f(i,j)=\frac{\sum_{i=0}^{|S|-1}|f(S(i))-f(S(i+1))|}{|S|} $$

另一种选取方式：假设在$t_0$下随机的生成一个状态序列，分别用$m_1$和$m_2$表示指标函数下降的状态数和指标函数上升的状态数，$\overline{\Delta f(i,j)}$表示指标函数增加的平均值。则$m_2$个状态中，被接受的个数为：

$$m_2\exp(-\frac{\overline{\Delta f(i,j)}}{t_0}) $$

则有平均接受概率：

$$P_0=\frac{m_1+m_2\exp(-\frac{\overline{\Delta f(i,j)}}{t_0})}{m_1+m_2} $$

令$P_0\rightarrow1$，反解$t_0=\frac{\overline{\Delta f(i,j)}}{\ln\left(\frac{m_2}{m_2P_0-m_1(1-P_0)}\right)}$。

温度的下降方法

  • 基本原则：下降的足够缓慢

1. 等比例下降

$$t_{k+1}=\alpha t_k, \ 0 < \alpha < 1 $$

2. 等值下降

$$t_{k+1}=t_k-\Delta t $$

每一温度下的停止准则

  • 在每个温度下要有足够的交换次数

固定长度方法：在每一个温度下，都使用相同的$L_k$（交换次数）。一般可以将$L_k$选择为关于问题规模$n$的一个多项式函数。

算法的终止原则

  • 基本原则：温度足够低

1. 零度法： $t < \epsilon$时结束

2. 循环总控制法：循环下降次数达到$K$时结束

3. 无变化控制法：在相邻的$n$个温度中得到的指标函数值无变化（一般来说两个温度相等就可以了）

组合优化问题：遗传算法

术语解释

生物进化中的概念 遗传算法中的应用
环境 适应函数，与求解的问题有关的函数
适应性 适应函数值（适应值）
选择：适者生存 以适应值的大小决定生存的概率，适应函数值大的解生存下来的概率大，适应函数值小的解生存下来的概率小
个体 问题的一个解
染色体 解的编码
基因 组成编码的元素
群体 被选定的一组解（以编码形式表示）
种群 根据适应函数选择的一组解（以编码形式表示）
婚配 以一定的方式由双亲产生后代的过程，在遗传算法中称作交叉操作
变异 编码的某些基因分量发生变化的过程

遗传算法

步骤：

(1) 给定群体规模$N$，交叉概率$P_c$和变异概率$P_m$，t=0；

(2) 随机生成$N$个个体作为初始群体；

(3) 对于群体中的每一个个体$x_i(i=1, 2, \dots, N)$分别计算其适应值$F(x_i)$；

(4) 如果算法满足停止准则，则转(10)；

(5) 对群体中的每一个体$x_i$依下式计算其选择概率：

$$P(x_i) = \frac{F(x_i)}{\sum_{j=1}^{N}F(x_j)} $$

(6) 依据选择概率，从群体中随机地选取$N$个个体，得到种群；

(7) 依据交叉概率$P_c$从种群中随机选择两个个体作为双亲进行交叉操作，其两个后代进入新的群体，种群中未进行交叉操作的个体，直接复制到新群体中；

(8) 依据变异概率$P_m$从新群体中选择个体进行变异操作，用变异后的个体代替新群体；

(9) 用新群体代替旧群体，$t=t+1$，转(3)；

(10) 进化过程中适应值最大的个体，作为算法得到的最优解输出；（注意这里是选择整个过程中适应值最大的个体，而非最后一代中适应值最大的个体）

(11) 结束。

一个例子

求解下列函数的最大值，其中$x$取值为区间$[0, 31]$上的整数

$$f(x)=x^2 $$
  • 适应函数

由于要求$f(x_i)$的最大值，因此可以将$f(x_i)=x_i^2$作为适应函数。

一般用$F(x_i)$表示适应函数，即

$$F(x_i)=f(x_i) $$
  • 编码问题

对可能的解进行编码，以便于可以用遗传算法求解。

二进制编码是一种常用的编码方法。

该例中$x$取值范围为$[0, 31]$间的整数，刚好可以用5位二进制表示，因此可以用5位二进制表示该问题的解，即染色体。

例如：00000表示$0$，11111表示$31$。编码选择的一个要求是：一一对应。

  • 选择操作

设$x_i(i=1, \dots, N)$是$N$个可能的解（个体），组成规模为$N$的群体，$F(x_i)$为解$x_i$的适应值。选择操作就从群体中可重复地选择$N$个解形成种群。

选择操作的基本原则：适应值越大的解被选中的概率越大。选择概率：

$$P(x_i)=\frac{F(x_i)}{\sum_{j=1}^NF(x_j)} $$

在实现过程中，有两种常用的选择算法：

  • “轮盘赌”选择算法

(1)$ r = random(0, 1)$（均匀分布）,$ s = 0, i = 0$;

(2) 如果$s > r$，则转(4);

(3)$ i = i + 1, s = s + p(x_i)$, 转(2);

(4) $x_i$即为被选中的染色体，输出$i$;

(5) 结束。

  • “确定性”选择算法

对于规模为$N$的群体，选择概率为$P(x_i)$的个体$x_i$被选中次数的期望$e(x_i)$为：

$$e(x_i)=P(x_i)N $$

对于群体中的每一个$x_i$，首先选择$\lfloor e(x_i)\rfloor$次，共得到$\sum_{i=1}^N\lfloor e(x_i)\rfloor$个个体，

然后按照$e(x_i)-\lfloor e(x_i)\rfloor$从大到小对染色体排序，依次取出$N - \sum_{i=1}^N\lfloor e(x_i)\rfloor$个个体，这样就得到了$N$个染色体。

“确定性”选择算法没有随机性，每个个体被选择的次数是一定的。

  • 交叉操作

交叉操作常用于两个双亲产生后代的过程。

设$a, b$是两个进行交叉操作的双亲染色体：

$$a = a_1a_2a_3\dots a_ia_{i+1}\dots a_n $$
$$b = b_1b_2b_3\dots b_ib_{i+1}\dots b_n $$

其中$a[i+1,n], b[i+1,n]$为交叉位。那么进行一次交叉操作以后，二者的后代为：

$$a = a_1a_2a_3\dots a_ib_{i+1}\dots b_n $$
$$b = b_1b_2b_3\dots b_ia_{i+1}\dots a_n $$
  • 变异操作

在二进制编码中，变异的基因由$0$变成$1$或者由$1$变成$0$。

进阶话题

编码问题

采用什么样的编码与具体问题有关，同时还要考虑表示精度以及如何进行交叉、变异操作。

例如：若$x$的取值为$[0, 31]$，精度可以有多个选择。

编码长度与精度之间的关系：

编码长度：$n$

取值范围：$[a, b]$

$n$位二进制数可以表示$0$到$2^n-1$共$2^n$个数

若以$\epsilon$表示允许的表示误差，那么一定有

$$\frac{b-a}{2^n-1} \leq \epsilon $$

故

$$n \ge \log_2(\frac{b-a}{\epsilon}+1) $$

编码与取值的对应关系

$$x_i = a + i \times\frac{b-a}{2^n-1} $$

即$x_0, x_1, \dots, x_{2^n-1}$的编码分别为$00\dots00, 00\dots01, \dots, 11\dots11$。

例：$x$取值范围为$[0, 31]$，精度$\epsilon=0.5$

$$n \ge \log_2(\frac{b-a}{\epsilon} + 1) \approx 5.98 \rightarrow 6 $$
$$x_i = i \times \frac{31}{63} = 0, 0.49, 0.98, 1.47, ..., 30.51, 31 $$

然而我们要求的精度是$0.5$，这是怎么回事呢？

如果严格要求间隔为$0.5$，那么$x_i = 0 + i\times0.5$。也就是说编码000000，000001，000010，…，111110对应$0, 0.5, 1, 1.5, \dots, 31$，而111111对应$31.5$（多余的）。

这时，我们可以令111111的适应值为$0$，在下一代选择中被淘汰。或者在交叉、变异操作中一出现111111就将其删除（并补充一个）。

适应函数

遗传算法求解适应函数的最大值，$\geq0$。有两种构造方式：

$$F(x)= \begin{cases} f(x), & f(x) > 0\\ 0, & Others \end{cases} $$

或

$$F(x)= \begin{cases} f(x) + M, & f(x) + M > 0 \\ 0, & Others \end{cases} $$

其中$M>0$为常数（超参数）

若要求最小值，可以求$F(x)=M-f(x)$的最大值，$M$是超参数

加速的思想

1. 非线性加速

$$F(x) = \begin{cases} \frac{1}{f_{max}-f(x)}, & f(x) < f_{max}\\ M, & Others \end{cases} $$

其中$f_{max}$为当前得到的最好结果，$M$是超参数

如果求$f(x)>0$的最小值，可以改为：

$$F(x) = \begin{cases} \frac{1}{f(x)-f_{min}}, & f(x) > f_{min}\\ M, & Others \end{cases} $$

2. 线性加速

$$F(x) = \alpha f(x) + \beta $$

原则：

适应函数与指标函数在群体内的平均值相等.

群体内最大的适应函数值是群体内指标函数平均值的$M$倍，其中$M$是一个比较大的常数.

于是有

$$\alpha \frac{\sum_{i=1}^N f(x_i)}{N} + \beta = \frac{\sum_{i=1}^Nf(x_i)}{N} $$
$$\alpha \max_{1\le i \le N}f(x_i) + \beta = M \frac{\sum_{i=1}^Nf(x_i)}{N} $$

3. 基于排序的适应函数

第$i$位个体的适应值直接设为$i$，选中概率为：

$$P(x_i) = \frac{i}{\sum i} = \frac{2i}{N(N+1)} $$

和适应函数的绝对大小无关，只和次序有关。

TypeScript中的异步

JavaScript语言是一门单线程语言，一次只能完成一个任务。若有多个任务执行，必须按照队列来执行，即前一个任务完成以后，再执行下一个任务。在这种模式下，只要有一个任务执行消耗了很长时间，后面的任务都无法执行，常见的浏览器无响应(假死)往往就是因为这样的原因而发生的。

为了解决这个问题，JS将任务执行模式分成同步和异步：

同步: 后一个任务等待前一个任务结束后再执行。程序的执行顺序和任务的排列顺序是一致的、同步的。

异步: 每一个任务有一个或多个回调函数(Callback)，前一个任务结束后，不是执行后一个任务，而是执行回调函数；后一个任务则不等前一个任务结束就执行。

  • 看个例子：
var a: number = 0;
var b: number = 0;
function foo() {
    a = 1;
}
setTimeout(foo, 1000);
b = a + 1;
console.log(b);

// 输出: 1

这其中的foo就是回调函数。所谓回调函数，就是被主线程放到任务队列中的代码，等到所有同步代码执行完成后才会被执行。

不过在Nest中，我们更常使用的工具不是回调函数，而是Promise与async, await。

  • Promise: Promise对象表示一个异步操作的最终结果，一个Promise对象的当前状态可以为Pending, Fulfilled, Rejected中的其中一个。与Promise对象交互的主要是then方法。该方法注册了两个回调函数，用于接收promise的终值或本promise不能执行的原因。
  • then方法: then方法接收两个可选参数: promise.then(onFulfilled, onRejected)。如果onFulfilled是函数，当promise执行结束后其必须被调用，且第一个参数为promise的终值；onRejected同理，第一个参数为promise的拒因。then方法最终返回一个promise对象。
async function foo() {
    return 'LTNS'
}
foo().then(console.log)

// 输出: LTNS
  • async/await: async与await实际是promise的语法糖
    • async: 函数前的async关键字意味着这个函数总返回一个promise值，即使你没有显式地那么写。就这么简单。
    • await: await关键字只能在async函数中使用，他可以让js等待，直到一个promise执行并返回它的结果，js才会继续往下执行。

map()

Array.map()是内置的TypeScript函数，用于创建新数组，并在此数组的每个元素上调用提供的函数。下文提到lodash时也会讲到。

基本用法为：

array.map(callback[, thisObject])
  • callback: 从当前元素的元素生成新Array元素的函数
  • thisObject: 执行回调时用作此对象的Object

例如：

let arr = [1, 2, 3];
const res = arr.map(x => x ** 2);
// res = [1, 4, 9]

另外，如果要在map中使用异步函数，需要像下面这样做

const res = await Promise.all(
	arr.map(async (x) => {
    return await SomeAsyncFunc(x);
  })
);

这里用到了Promise.all方法，它接收一个可迭代的包含Promise的容器(容器，hhh)，并返回一个单一的Promise对象。如果可迭代对象内所有Promise的状态都为Fullfilled，那么Promise.all返回的Promise对象的状态也为Fulfilled，并且带有这个异步函数的返回值列表。

关于Lodash，我们最主要用的还是它关于Array和String的方法。比如说

  • _.concat(array, [values])

创建一个新数组，将array与任何数组 或 值连接在一起。例如：

let array = [1];
let other = _.concat(array, [2, 3, 4]);
// other = [1, 2 ,3, 4]
  • _.filter(collection, [predicate=_.identity])

遍历 collection（集合）元素，返回 predicate（断言函数）返回真值 的所有元素的数组。 predicate（断言函数）调用三个参数：(value, index|key, collection)。例如：

var users = [
  { 'user': 'barney', 'age': 36, 'active': true },
  { 'user': 'fred',   'age': 40, 'active': false }
];
 
_.filter(users, function(o) { return !o.active; });
// => objects for ['fred']
 
// The `_.matches` iteratee shorthand.
_.filter(users, { 'age': 36, 'active': true });
// => objects for ['barney']
 
// The `_.matchesProperty` iteratee shorthand.
_.filter(users, ['active', false]);
// => objects for ['fred']
 
// The `_.property` iteratee shorthand.
_.filter(users, 'active');
// => objects for ['barney']

NestJS是什么?

  • Nest(NestJS)在Node.js框架之上提高了一个抽象级别，提供了一个开箱即用的应用程序架构，允许开发人员团队创建高度可测试、可扩展、松耦合且易于维护的应用程序。

这段话虽然简单，但也阐述了NestJS的主要特点，它解决的是Node.js生态开发中的架构问题，也就是说使用它，你就就会获得不错的后端架构。而且它还表明了这个架构的特点，高度可测试、可扩展、松散耦合易于维护。 NestJS还有一些特点，它使用渐进式JavaScript，内置并完全支持TypeScript(但你也可以使用纯JS来进行开发)并结合了OOP(面向对象编程)，FP(函数式编程)和FRP(函数式响应编程)的元素。这些特点保证了NestJS是一个高度可扩展和松散耦合度的框架。也就我们开发时常说的高内聚低耦合

TypeORM

什么是ORM?

ORM是Object Relational Mapping的缩写，译为“对象关系映射”，它解决了对象和关系型数据库之间的数据交互问题。

简单理解ORM的作用：定义一个对象，这个对象对应着一张表；这个对象的一个实例，就对应着表中的一条记录。这样做的好处就在于我们可以通过对象的形式操作数据库。

ORM的优缺点:

  • 方便维护，代码量少，工具多，自动化能力强
  • 性能不如传统SQL语句效率高：因为SQL语句的形式是可以优化的，而ORM工具的高度集成化的形式，没法进行SQL语句的优化。

安装TypeORM

npm install --save @nestjs/typeorm typeorm mysql2

Entity

对应于MySQL中的数据，我们需要为表中的数据编写实体，例如:

@filename('base.ts')
import {
  CreateDateColumn,
  DeleteDateColumn,
  PrimaryGeneratedColumn,
  UpdateDateColumn,
} from 'typeorm';

export abstract class Base {
  @PrimaryGeneratedColumn()
  id: number;

  @CreateDateColumn({ type: 'datetime' })
  created_at: number;

  @UpdateDateColumn({ type: 'datetime' })
  updated_at: number;

  @DeleteDateColumn({ type: 'datetime' })
  deleted_at?: number;
}

@filename('SomeEntity.ts')
import { Entity } from 'typeorm'
import { Base } from './base'

@Entity()
export class SomeEntity extends Base {
  // Definitions
}

可以看到，这里先实现了一个Base类作为其他所有实体的抽象基类，提供一些所有Entity都具有的属性。当然也可以不这么做。

增删查改

一般我们使用TypeORM中的QueryBuilder进行SQL查询。它允许你使用优雅便捷的语法构建SQL查询，执行并获得自动转换的实体。

查：

const user = await getRepository(User)
	.createQueryBuilder("user")
	.where("user.id = :id", { id: 1 })

增：

await getRepository(User)
	.createQueryBuilder()
	.insert()
	.into(User)

改：

await getRepository(User)
	.createQueryBuilder()

	.set({ key: value })
	.where("id = :id", { id: 1 })

删：

await getRepository(User)
	.createQueryBuilder()

	.from(User)

假设有以下的实体：

import { Entity, PrimaryGeneratedColumn, Column, OneToMany } from "typeorm";
import { Photo } from "./Photo";

@Entity()
export class User {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  name: string;

  @OneToMany(type => Photo, photo => photo.user)
  photos: Photo[];
}

import { Entity, PrimaryGeneratedColumn, Column, ManyToOne } from "typeorm";
import { User } from "./User";

@Entity()
export class Photo {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  url: string;

  @ManyToOne(type => User, user => user.photos)
  user: User;
}

现在我们要在查询User表的时候，同时获取/筛选它的photos信息，那么我们可以这么做：

const user = await createQueryBuilder("user")
  .leftJoinAndSelect("user.photos", "photo")
  .where(/* Some Conditions on photos */)
  .getMany();

可以和Django ORM处理外键的方式进行对比。

需要注意的是，在表中数据庞大的情况下，最好尽可能地先进行条件筛选，再进行联查。不然联查的过程中可能会连接一个非常庞大的表，这会导致效率严重下降。

SQL慢查询

生产过程中，在Mentor的带领下解决了几个慢查询的问题。总结一下情况大概有这么几个：

  • 前文提到的联查：数据量庞大时，先尽可能地进行条件筛选，再进行联查。

  • getMany()和getCount()的效率差异是巨大的，所以如果某个功能只要求计数的话，最好不要getMany()再筛选，宁愿多几条筛选的SQL语句。

  • 关于SQL中的like：在数据量不大的时候还是挺好用的。但是数据量庞大的时候，最好还是建立索引，使用

MATCH (column) AGAINST (pattern in BOOLEAN MODE);
  • （网上看见的，不知真假）最好不要使用or进行查询，如果可以的话，使用union连接两次查询的结果。
    • 然而TypeORM的QueryBuilder并不支持union，所以只能使用or和orwhere了。

一

前天凌晨，我被计系概整到破防。躺在床上思考自己逐渐丧失的交流欲望，是压力导致的吗，还是性格使然？当我感到焦虑时，我不想与任何人说任何事；然而焦虑却又如潮水般袭来，所以我宁愿把自己封闭起来。

想着这些便不知不觉进入了梦乡。在梦里，我向你诉说着这一年来你不知道的事，关于我分流到了计算机系，关于我后悔在清华里做的每一个决定。我们紧挨着坐在南十北昏暗的走廊里，当我刚说完这一切，泪水就抢先于你的声音出现在梦境世界里。你知道我一到冬天就手脚冰凉，但我们在梦里相拥时，我却第一次感受到接触的温暖，脸颊对脸颊，比我那炙热的心还要烫一些。

第二天是万圣节，直到晚上我才想起10月还有31日。为了证明我还活着，在22：00时我毅然决然冲向校外，即使我压根不知道我此行的目的地何在。当我绕着五道口绕了一圈回来时，却戏剧性地发现北京健康宝弹窗了，遂折磨辅导员将近两小时。当时他们要我做好在外过夜的准备，可弹窗之人哪里去得了什么酒店呢。于是我想，倘若真的没法儿回清华，我就打电话叫你出来喝酒，只可惜社区办事的效率竟还算合理，一个小时内便消除了我的弹窗。虽在外吹了一晚上冷风，四肢僵劲不能动，但我却感到意外的快乐——只因这场本无期望的出行居然为我的生活增添了一抹别的色彩。
